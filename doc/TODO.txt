0. Prerequisites
Add NotReady value to PiSubmarine::Error::Api::ErrorCondition.
This value represents a generic event where a resource or operation did not fail, but is temporarily unavailable. The caller may retry later.

Place all code in app folder. src is not needed for now and can be cleared. We will decide later what is worth exporting from app to src.

I. Functionality:

The GUI can be split in these functional groups:
1) Video
2) Input
3) Telemetry

Video can be split further into:
1) Maintain video lease 
2) Maintain video subscription (tell the server where to send RTP packets)
3) Maintain gstreamer pipeline
4) Render video onto GPU-backed QML item

Input can be split:
1) Maintain Input lease.
2) Read user input
3) Send user input to drone Input endpoint

Telemetry can be split:
1) Maintain telemetry lease (single shared lease for all telemetry providers)
2) Maintain telemetry subscription (tell the server where to send telemetry. Shared for all providers)
3) Receive Telemetry packets
4) Render gauges in QML

II. Threads
1) UI thread - QML rendering, view state changes, user input capturing
2) Controller thread - calling PiSubmarine domain-specific interfaces, listening to UI state changes, emitting signals to UI upon domain-specific interfaces state changes.
3) Lease thread - gRPC lease client is blocking, so it should be separate from both Controller and UI threads.

III. Lease

1. IAsyncLeaseIssuer as a station-side asynchronous wrapper around Lease.Api.ILeaseIssuer.

2. Lease thread worker that implements IAsyncLeaseIssuer and forwards lease requests from Controller thread to Lease.Api.ILeaseIssuer that works in Lease thread.

Dependency-injections: 

- Lease.Api.ILeaseIssuer (in practice will be Lease.Client.Grpc)

3. SyncLeaseIssuerProxy implements Lease.Api.ILeaseIssuer and acts as a synchronous polling facade over IAsyncLeaseIssuer. If a valid lease is not available yet, it returns ErrorCondition::NotReady instead of blocking. SyncLeaseIssuerProxy must remember pending lease requests. If AcquireLease/RenewLease is called repeatedly while the previous async request is still pending, it must not enqueue a new async request every time. It should return ErrorCondition::NotReady until the pending request completes. If a valid cached lease exists, return it immediately.

4. Fake lease provider that implements Lease.Api.ILeaseIssuer and locally grants every lease.

VI. Input

1) Input Controller - app/PiSubmarine/Operator/Station/Input/Controller.cpp

Reads user input from interactive QML widgets (knobs, toggles, etc), keyboard or joystick, forms OperatorCommand (see Control.Api) and sends it to the drone via Control::Api::Input::ISink instance.

Dependency-injections:

- Control.Api.Input.ISink (in practice will be instance of Control.Client.Udp).
Input lease is maintained internally by Control.Client.Udp. In production, Control.Client.Udp receives SyncLeaseIssuerProxy as its Lease.Api.ILeaseIssuer dependency.

Listens to ViewModel signals:

- Key strokes, joystick movements, interactive widgets changes

2) Input View

app/PiSubmarine/Operator/Station/Input/View/*

Defines Input intent ViewModel, emits change signals.
Defines QML widgets for interactive user input (toggle lamps, toggle auto-pilot, etc).

Captures axis inputs from joystick or keyboard for movement control.

VII. Layout

Main Window layout defines how input, telemetry and video QML items are drawn relative to each other.

VIII. Composition Root

int main() is the composition root.

It is responsible for:

Creating QGuiApplication / QApplication.
Parsing CLI arguments.
Deciding whether to create real or fake implementations.
Creating PiSubmarine domain implementations.
Creating Controllers.
Creating ViewModels.
Creating QML engine and registering/exposing ViewModels and QML item types.
Creating thread handlers / QThreads.
Moving Controllers and workers to the correct threads.
Creating tick sources, such as QTimer-based tick sources.
Wiring signals and slots between ViewModels, Controllers, lease worker, and QML.
Starting worker threads.
Loading the main QML layout.
Starting the Qt event loop with app.exec().
Performing orderly shutdown when the application exits.

main() decides which implementation to instantiate.

For example:

real video controller or fake video controller
real telemetry controller or fake telemetry controller
real lease client or fake lease issuer
real input sink or fake input sink

Fake and real implementations should expose the same controller-facing interfaces where possible.

The UI thread remains the Qt main thread. QML, ViewModels, and QQuickItems live there.

Controller objects should live in the Controller thread.

The blocking Lease.Client.Grpc object should live in the Lease thread.

All cross-thread communication should use queued Qt signal/slot connections or another explicitly thread-safe mechanism.

main() should not implement application logic. It should only construct, configure, wire, start, and shut down the application.
